# 検証してから完了（Verify Before Done）

出典: obra/superpowers `verification-before-completion` を kskphotos 向けに軽量化。

## 鉄則

```
検証コマンドを「今このターンで」実行していないなら、通過したと主張してはいけない
```

「直したはず」「たぶん大丈夫」「動くと思う」は**完了報告ではない**。証拠が先、主張は後。

## ゲート手順

完了・満足・成功を口にする前に必ず:

1. **特定** — その主張を証明するコマンドは何か
2. **実行** — フルで新規に走らせる（前回の結果を流用しない）
3. **読む** — 出力全体・終了コード・失敗件数を確認
4. **照合** — 出力が主張を裏付けるか。否なら実状を証拠付きで述べる
5. **その後だけ** — 証拠と共に主張する

## kskphotos の検証コマンド

| 主張 | 必要な証拠 | 不十分なもの |
|---|---|---|
| テスト通過 | `npm --prefix app run test:run` → 0 failures | 前回の実行・「通るはず」 |
| 型/ビルドOK | `npm --prefix app run build` → exit 0 | Lintが通った（Lint≠型/ビルド） |
| バグ修正 | 元の症状を再現するテストが pass | コードを変えた＝直った、ではない |
| 要件充足 | 計画/Issueを1行ずつチェックリスト照合 | テストが通った＝完了、ではない |
| Terraform | `terraform plan` の差分を読む（apply は -target 必須） | plan を見ずに apply |
| 機能の「完了」 | **docs/ 更新まで含めて完了**（CLAUDE.md規約） | コードだけ動いた |

## レッドフラグ（STOP）

- 「should / probably / seems」を使っている
- 検証前に「完璧!」「できた!」と満足を表明
- 検証せずに commit / push / PR / Codexレビュー依頼へ進もうとしている
- サブエージェントの「成功しました」を鵜呑みにしている（VCS diff を自分で確認せよ）
- 疲れて早く終わらせたい

## 合理化への反論

| 言い訳 | 現実 |
|---|---|
| 「もう動くはず」 | 検証を実行せよ |
| 「自信ある」 | 自信≠証拠 |
| 「今回だけ」 | 例外なし |
| 「Lint通った」 | Lintは型/コンパイルを見ない |
| 「エージェントが成功と言った」 | 自分で diff を確認 |

## 底線

コマンドを走らせ、出力を読み、**それから**結果を述べる。非交渉。
