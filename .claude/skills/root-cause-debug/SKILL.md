# 根本原因デバッグ（Systematic Debugging）

出典: obra/superpowers `systematic-debugging` を kskphotos 向けに軽量化。

## 鉄則

```
根本原因の調査を終える前に、修正を提案してはいけない
```

場当たり修正は新たなバグを生み、症状を隠す。**症状を直す＝失敗**。簡単に見えるバグにも根本原因はある。

## 4フェーズ（順に完了させる）

### フェーズ1: 根本原因の調査（修正の前に必ず）
1. **エラーを丁寧に読む** — スタックトレース全体、行番号・パス・コードを飛ばさない。答えが書いてあることが多い
2. **確実に再現** — 毎回起きるか、手順は何か。再現しないならデータを集める（推測しない）
3. **最近の変更を確認** — `git diff` / 直近コミット / 依存・設定の変化
4. **多層系は境界で証拠を集める** — 例: API→service→DB、CI→build→deploy。各境界で「入る値/出る値」をログ化し、**どこで壊れるか**を先に突き止める
5. **データフローを遡る** — 悪い値の発生源まで遡り、症状ではなく源で直す

### フェーズ2: パターン分析
- 同じコードベースの**動いている類似例**を探す
- 参照実装は**全部読む**（飛ばし読み禁止）
- 動く版と壊れた版の**差分を全部**列挙（「これは関係ない」と決めつけない）

### フェーズ3: 仮説と検証（科学的手法）
- 仮説を1つだけ明文化: 「Yだから X が根本原因だと考える」
- **最小の変更**で検証（1度に1変数）
- 効かなければ**新しい仮説**へ。上に修正を積み増さない

### フェーズ4: 実装
1. **失敗するテストを先に作る**（→ `tdd-vitest` / `npm --prefix app run test:run`）
2. 根本原因への**単一の修正**（「ついでにリファクタ」禁止）
3. 検証: テスト通過・他が壊れていない・問題が実際に解決
4. **3回直して効かなければ STOP** → アーキテクチャを疑う（人に相談）。それは仮説の失敗ではなく設計の誤り

## レッドフラグ（→ フェーズ1へ戻る）
- 「とりあえず直して後で調査」「Xを変えて様子見」
- 調査前に解決策を列挙している
- 修正を重ねるたび別の場所に新しい問題が出る（=設計の問題）

## 合理化への反論

| 言い訳 | 現実 |
|---|---|
| 「簡単だから手順不要」 | 簡単なバグにも根本原因。手順はむしろ速い |
| 「緊急で時間がない」 | 体系的デバッグは当て推量の試行錯誤より速い |
| 「見えたから直す」 | 症状が見える ≠ 根本原因の理解 |

## 効果（出典の実測）
体系的: 15〜30分・初回修正率95%・新規バグほぼ0 / 場当たり: 2〜3時間・40%・頻発。
