# 17. 管理画面：保守運用契約書ジェネレータ

## このドキュメントの目的

`/admin/maintenance` の保守運用契約書作成機能の概要・使い方・設計判断を記録します。制作の[業務委託契約書](./15-admin-contract-generator.md)とは別契約として扱う理由もここに記します。

## 概要

公開後の **保守・運用（更新代行・ホスティング維持）** の契約書を、フォーム穴埋め → 整形プレビュー → **印刷（PDF 保存）** で発行する管理者向けツール。制作の業務委託契約が「作って納める」一回性の契約なのに対し、こちらは **月額・継続** の契約で、相手から「運用代行の書類を」と求められた際に渡す。

| 項目 | 内容 |
|------|------|
| ルート | `/admin/maintenance`（認証必須・`robots noindex`）|
| 出力方式 | **印刷 → PDF**（ブラウザの「PDF に保存」）。追加ライブラリ・サーバー生成なし |
| 保存 | **都度生成のみ**（DB なし）。フォーム状態はクライアントのみ |
| 構成 | `page.tsx`（サーバ）＋ `maintenance-generator.tsx`（クライアント） |

## なぜ制作契約と分けるか（設計判断）

- **契約の性質が違う**：制作は成果物の完成・納品・著作権が論点。運用は継続的役務で、月額・対応範囲（SLA）・解約・自動更新が論点。1つの契約書に混ぜると条項が肥大化し、解約や更新の効力範囲が曖昧になる。
- **相手の要求に対応**：発注者が「仕様書・運用代行の書類」を別途求めるケースがある。制作（業務委託契約書）＋運用（本書）＋設計書/仕様書 を揃えると過不足ない。
- 出力方式・DB 非保存・印刷時の挙動（`print:hidden`／`.contract-paper`／`@page margin:16mm`）は契約書ジェネレータと共通。詳細は[15章](./15-admin-contract-generator.md)。

## 条項の構成

フォーム入力で差し替え、チェックで条項を出し分け（条番号は自動採番）。

| 条 | 内容 | ポイント |
|----|------|---------|
| 目的 | 対象サイト（名称・URL）の保守運用を委託 | |
| 業務内容 | サーバー/SSL/ドメイン維持、不具合対応、軽微な更新、ライブラリ更新 | 対応範囲（更新回数等）をフォームで指定 |
| 対応範囲外 | 新規ページ・デザイン刷新・機能追加・撮影は別途見積り | **スコープ膨張の防止** |
| 委託料・支払 | 月額固定・支払サイクル・振込先 | 支払いは「柔軟に対応・基本は銀行振込」で統一 |
| 無償サポート期間 | 制作→運用移行時の最初の N ヶ月無償（任意） | 月額は無償期間満了の翌月から発生 |
| 対応時間・連絡方法 | 連絡手段＋一次回答の目安（簡易 SLA） | **24/365 即時対応は保証しない**を明記 |
| 第三者サービスの利用 | クラウド障害・仕様変更・終了の影響と免責 | |
| ドメイン・データの所有 | ドメイン/コンテンツ/データは**甲に帰属** | |
| バックアップ | 取得方針＋第三者障害時のデータ消失免責 | |
| 契約期間・更新 | 起算日から N 年・自動更新（予告なき場合） | |
| 中途解約 | N ヶ月前の書面通知で解約・当月分は返金なし | |
| 契約終了時の引き継ぎ | サイトデータ・管理情報の引き渡し | |
| 秘密保持／個人情報／反社排除 | 共通条項 | ログイン情報も秘密保持の対象 |
| 損害賠償・免責 | 賠償上限＝**直近 N ヶ月分の月額運用料** | 継続役務に応じた上限設定 |
| 譲渡制限／変更／合意管轄 | 共通条項 | 月額・対応範囲の変更は書面合意 |

- **政治・選挙特約**：候補者サイトの運用向け。公選法順守、選挙運動用文書図画の更新は告示〜投票前日まで・投票日当日は更新しない等。チェックで ON/OFF。

## 使い方

1. 管理画面 → **保守運用** を開く。
2. 左フォームに発注者・対象サイト・月額・対応範囲・期間などを入力。右側プレビューに即反映。
3. **制作→運用移行（3ヶ月無償サポート）** と **政治・選挙特約** はチェックで ON/OFF。
4. 「**印刷 / PDF 保存**」→ 印刷ダイアログで「**PDF に保存**」を選択してダウンロード。

## 注意・今後

- 出力はあくまでひな型。重要・高額・継続契約は専門家確認を推奨（本ツールは法的助言ではない）。
- 入力値の保存（DB 化）、見積書連携、仕様書/要件定義書ジェネレータ化は将来の拡張候補。
